## 技术面叙述

### 1. UP 区域的“屏蔽用户名关键词”组件形态不符合用户预期

现象：

截图中，“标题”文本框下方是标准的 `复制` / `屏蔽` 操作行；UP 文本框下方先有 `复制` / `屏蔽用户 UID: 440237866`，再额外出现一个独立的 `屏蔽用户名关键词: 摔哥秀` 组件。用户指出“不需要此”，这里指的是这个额外独立组件形态，而不是否定 UP 文本框可以创建用户名关键词规则。

技术转译：

- UP 文本框仍应支持创建用户/UP关键词规则。
- 但 UI 不应使用独立的 `屏蔽用户名关键词: <当前值>` 长按钮组件。
- UP 文本框应像“标题”文本框一样，在当前字段操作行里提供一个普通 `屏蔽` 按钮。
- 该 `屏蔽` 按钮服务于 UP 文本框的当前编辑值，规则类型仍是用户/UP关键词。
- UID 屏蔽按钮仍保留，并继续使用原始 UID，而不是编辑后的 UP 文本。

验收方向：

- UP 区域不再出现独立的 `屏蔽用户名关键词: 摔哥秀` 组件。
- UP 文本框下方出现与标题区域一致的普通 `屏蔽` 操作。
- 点击 UP 区域的 `屏蔽` 时，创建用户/UP关键词规则。
- 如果用户编辑 UP 文本框，点击 `屏蔽` 应使用编辑后的文本。
- 点击 UID 屏蔽仍使用原始 UID `440237866`。

### 2. 推荐理由尚未完整纳入屏蔽体系

现象：

用户明确要求：出现推荐理由时，推荐理由的屏蔽词需要纳入屏蔽规则中的推荐理由分类，同时推荐理由也要纳入屏蔽系统。

技术转译：

- 推荐理由不应只是弹窗里可复制或临时展示的文本。
- 推荐理由需要作为独立输入字段进入屏蔽匹配上下文。
- 规则模型或分类展示中需要有清晰的“推荐理由”归类。
- 推荐理由命中后，应能影响推荐内容过滤结果。

验收方向：

- 推荐理由文本可被创建为屏蔽规则。
- 新增规则在设置页中进入“推荐理由”分类，而不是混入标题、用户关键词或旧规则兼容分类。
- 推荐流过滤时，推荐理由字段参与匹配。
- 有推荐理由命中的卡片按预期被屏蔽。

### 3. 设置页存在过时分类入口

现象：

用户认为“精确文本分类页”和“旧规则兼容分类页”已经失去存在意义，需要删除。

技术转译：

- 设置页不应继续暴露已经完成迁移或不再承担用户任务的分类页面。
- 删除入口不等于删除底层兼容能力。
- 如果旧规则兼容仍用于数据迁移或后台解释，应保留内部逻辑，但从正常设置导航中移除。

验收方向：

- 设置页中不再显示“精确文本分类页”。
- 设置页中不再显示“旧规则兼容分类页”。
- 用户无法通过正常入口进入这两个页面。
- 旧数据读取、迁移、去重和屏蔽运行行为不因 UI 删除而回退。
